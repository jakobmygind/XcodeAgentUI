# AGENTS.md тАФ XcodeAgentUI

Project conventions, patterns, and decision log for AI agents and contributors.

## Project Overview

Native macOS control center for autonomous Xcode Agent Runner instances.
Built with SwiftUI + Point-Free dependency ecosystem (`swift-dependencies`, `swift-sharing`, `swift-navigation`).

## Conventions

### Architecture Patterns

| Pattern | Where | Notes |
|---------|-------|-------|
| `@Observable` + `@MainActor` | All service classes | Thread-safe reactive state |
| `@DependencyClient` | `Dependencies/` folder | Macro-generated clients for DI |
| `@Dependency` | Service classes | Property-wrapper injection |
| `@Shared(.appStorage)` | Preferences / settings | Observation-aware persistence |
| `withDependencies` | All test `setUp` methods | Override deps per-test |

### File Organization

- **`Dependencies/`** тАФ `@DependencyClient` structs (one per file, named `*Client.swift`)
- **`Services/`** тАФ Business logic classes (`@Observable @MainActor`)
- **`Models/`** тАФ Value types, enums, domain models
- **`Views/`** тАФ SwiftUI views
- **`Mocks/`** тАФ Preview and test fixture data

### Naming

- Dependency clients: `FooClient` (struct in `Dependencies/FooClient.swift`)
- Services: `FooManager` or `FooStore` (class in `Services/`)
- Tests: `FooTests` in `XcodeAgentUITests/`

## How to Add a New Dependency Client

1. Create `Dependencies/NewClient.swift`:

```swift
import Dependencies
import Foundation

/// Brief description of what this client does.
@DependencyClient
struct NewClient: Sendable {
  // Void-returning endpoints need no default
  var doSomething: @Sendable () -> Void
  // Non-Void, non-Optional endpoints need an explicit default
  var fetchCount: @Sendable () -> Int = { 0 }
  // Optional endpoints default to nil automatically
  var loadData: @Sendable (_ key: String) -> Data?
}

extension NewClient: DependencyKey {
  static var liveValue: NewClient {
    NewClient(
      doSomething: { /* real implementation */ },
      fetchCount: { /* real implementation */ },
      loadData: { /* real implementation */ }
    )
  }
  // testValue is auto-generated by @DependencyClient:
  // - Void endpoints become no-ops that call reportIssue
  // - Non-Void endpoints return the default and call reportIssue
}

extension DependencyValues {
  var newClient: NewClient {
    get { self[NewClient.self] }
    set { self[NewClient.self] = newValue }
  }
}
```

2. Use in a service:

```swift
@ObservationIgnored @Dependency(\.newClient) var newClient
```

3. Override in tests:

```swift
sut = withDependencies {
  $0.newClient.fetchCount = { 42 }
} operation: {
  MyService()
}
```

## Testing Patterns

### Always use `withDependencies`

Every test that creates an object with `@Dependency` properties must wrap
creation in `withDependencies`. This ensures deterministic behavior and
catches unintended dependency access via `reportIssue`.

```swift
override func setUp() {
  super.setUp()
  sut = withDependencies {
    $0.keychainClient = KeychainClient(
      save: { _, _ in true },
      load: { _ in nil },
      delete: { _ in true },
      hasValue: { _ in false }
    )
  } operation: {
    ProviderStore()
  }
}
```

### Override only what you need

With `@DependencyClient`, unoverridden endpoints call `reportIssue` in tests.
This is intentional тАФ it surfaces unexpected dependency usage. Override only
the endpoints your test exercises.

### XCTest + @MainActor

All test classes use `@MainActor` since the services under test are main-actor
isolated. Use `async` test methods when testing async code paths.

## Decision Log

### 2026-03-26 тАФ Adopt `@DependencyClient` macro for all DI clients

**Decision:** Replace manual `DependencyKey` implementations with
`@DependencyClient` macro from swift-dependencies.

**Why:**
- Eliminates boilerplate memberwise initializers
- Auto-generates `TestDependencyKey` conformance with `reportIssue` stubs
- Unoverridden endpoints in tests produce clear failure messages instead of
  silently returning defaults
- Aligns with current Point-Free best practices

**Trade-offs:**
- Requires swift-dependencies 1.1+ (already on 1.6+)
- `Bool`-returning endpoints need explicit defaults (macro can't infer)
- Slightly more opaque тАФ macro-generated code isn't visible in source

### 2026-03-26 тАФ `withDependencies` required in all tests

**Decision:** All test `setUp` methods must create subjects via
`withDependencies`, even if the test doesn't exercise dependency endpoints.

**Why:** Prevents silent test pollution from global dependency state and
ensures `@DependencyClient`'s `reportIssue` stubs catch unintended access.

## Rules

- **Always update README.md** when changing user-facing features, configuration,
  or installation steps.
- **Document learnings and decisions** in this file (AGENTS.md) when making
  architectural choices or establishing new patterns.
- **One client per file** in `Dependencies/` тАФ keep the folder scannable.
- **No singletons for testable code** тАФ use `@Dependency` instead. Singletons
  (like `NotificationManager.shared`) are acceptable only for system framework
  delegates that require it.
